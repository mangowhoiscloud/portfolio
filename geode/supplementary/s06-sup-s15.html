CoalescingQueue가 구조적으로 불필요한 증명 + C3/H2 Race Trace

CoalescingQueue 250ms dedup이 발화할 수 없는 이유

while(tool_use) 동기 루프 타이밍

LLM call

2-30s

tool exec

0.1-60s

dedup window

0.25s

동기 루프에서 같은 키로 재제출하려면 LLM call(2s+) + tool exec(0.1s+) = 최소 2.1초가 필요합니다. 250ms 윈도우 안에 재제출이 도착하는 것은 구조적으로 불가능합니다.

실측 검증

CoalescingQueue의 dedup 콜백이 실제로 발화한 횟수: 0회. 전체 세션 이력에서 한 번도 dedup이 작동한 적이 없습니다. callback 자체도 lambda _k, _d: None (no-op).

C3 + H2 Race Condition Traces

C3: Scheduler jobs.json 경합

T1: read(jobs.json) → job_list = [A, B]

T2: read(jobs.json) → job_list = [A, B]

T1: write([A, B, C]) # C 추가

T2: write([A, B, D]) # C 유실!

해결: fcntl.flock(LOCK_EX). 파일 수준 배타 잠금.

H2: announce 중복 발행

T1: check _active[key] → False

T2: check _active[key] → False

T1: set _active[key]=True, announce()

T2: set _active[key]=True, announce() # 중복!

해결: Lane.try_acquire()/manual_release() atomic check-and-set.

SessionLane이 두 결함을 동시에 해소하는 이유: per-key Semaphore(1)이 같은 키의 동시 접근을 원천 차단합니다. C3은 같은 jobs.json에, H2는 같은 session key에 대한 동시 접근이었습니다.

S15 보강: dedup 발화 0회(구조적 불가능). C3 jobs.json race + H2 announce race → SessionLane per-key Sem(1) 해소
