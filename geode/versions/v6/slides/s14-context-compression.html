HE-1 DEEP DIVE — 실행 흐름 ⑤

Context Compression: 80% WARNING → 95% CRITICAL → 3-Phase 압축

컨텍스트 윈도우가 넘치면 루프가 강제 종료됐다 — Lossless Selection으로 정보 손실 없이 비용 0.03%로 절감

컨텍스트 압축 3-Phase

Phase 1 — MONITOR

컨텍스트 사용률 상시 추적. 80% 도달 시 CONTEXT_WARNING 이벤트 발화 — 루프는 계속 진행하되 compact 준비 신호를 핸들러에게 전달한다.

↓

Phase 2 — LOSSLESS SELECT

Lossy 요약 대신 LLM이 코드로 관련 청크만 선별한다. 요약이 아니라 선택이기 때문에 정보 손실 0% — LLM 호출 비용 0.03%로 절감. 깊이 3이면 서브에이전트 독립 200K window로 유효 컨텍스트 600K 선형 확장.

Lossy 요약

구현 쉽지만 정보 소실 불가피

Lossless Selection

관련 청크만 선별 — 내용 유지

↓

Phase 3 — EMERGENCY PRUNE (95%)

95% 초과 시 CONTEXT_CRITICAL 이벤트 — 강제 pruning 발동. 최초 사용자 메시지 보존 + bridge 메시지 삽입 후 루프 재개.

Trade-off: 정보 손실 vs 크래시 방지

Lossy 요약은 구현이 쉽지만 정보가 사라진다. Lossless Selection은 LLM 호출 비용이 있지만 0.03%의 비용으로 정보 손실을 막았다.
